# TeamMaker — продукт целиком

Этот файл описывает TeamMaker на уровне всего продукта. Если ты Claude Code
и работаешь в конкретном репозитории (`web`, `ue`, `mobile`) — там будет
свой собственный `CLAUDE.md` с деталями именно этой части. Этот файл —
для общей картины и для людей, которые впервые смотрят на организацию.

## Что такое TeamMaker
Платформа для организации спортивных турниров. Модульная архитектура —
каждый вид спорта свой модуль. Старт — настольный теннис.

## Стратегия: web сначала, mobile — цель
Конечный продукт — нативное мобильное приложение, так как основная
аудитория (спортсмены, организаторы) пользуется телефонами. Но начинаем
с веб-версии: это быстрее и дешевле для проверки продуктовых гипотез
(турниры, сетки, запись участников и т.п.) до того, как вкладываться
в нативную разработку. web не финальная форма продукта, а шаг к mobile.

## Из чего состоит продукт (репозитории организации ProTeamMaker)

### `ProTeamMaker/web`
Веб-приложение — тестовый полигон для MVP. Регистрация, профиль спортсмена,
клубы, турниры, сетки. Цель этого репозитория — быстро проверить продуктовые
гипотезы и механику (турниры, сетки, запись), прежде чем инвестировать
в нативное мобильное приложение. Это не финальная форма продукта.
Статус: MVP в разработке, стек пока не выбран.

### `ProTeamMaker/mobile`
Нативное мобильное приложение — целевой основной продукт. Аудитория
(спортсмены, организаторы турниров) в основном пользуется телефонами,
поэтому конечная цель — здесь, а не в вебе. Создаётся после того как
гипотезы проверены на `web`, переносит и расширяет ту же функциональность.
Статус: не создан, ждёт результатов с `web`.

### `ProTeamMaker/ue`
Идея тренировочного симулятора на Unreal Engine с 3D-аватарами.
Сейчас не в активной разработке — отложено, приоритет ниже web/mobile.
Код существует (перенесённый старый проект), но развития пока не планируется.
Статус: WIP / на паузе.

## Принцип организации продукта
Все репозитории независимы технически — разный стек, разный цикл релизов,
разные команды/разработчики могут вестись параллельно без конфликтов.
Связь между ними — только на уровне продукта и общей цели, не на уровне кода.

## Кто работает над продуктом
Один человек на старте, разные части могут развиваться параллельно
по мере роста команды.
